# 产出单页PRD `/lny-prd-page`

- **极简模板**：本文 **「single_page_prd.md 模板」** 嵌入块（**含第 3 节顶整页 ASCII 线框占位**）。字段与规则见 **「单页 PRD 产出具约束」**。

- 接口需求（优先新结构）：`api_spec.md` §4 索引 + `api/API-*.md` 明细
- UI 事实来源：**`ui/PAGE-*.md` / `ui/COMP-*.md`**（布局与组件详述）；**`ui_manifest.md`** 仅 **§3/§4 索引** 与 **§5 全局说明**。
- 出页与跳转：`ui_manifest.md`/`ui/`（合法 **PAGE-***）；读 `main_spec.md` **1.2** 时 **仅**作第 2 节背景（见 **产出具约束 → 2.**）
- **桌面端跳转清单**：**不**写业务壳换页；见 **产出具约束 → 4.**

## 单页 PRD 产出具约束

> **禁止**把本节说明文字写入 `pages_prd`；落盘 **仅** 模板骨架 + **取值**（表内单元格、列表项、`###` 分区按需 **复制增删**）。
> **禁止**为 **`lny-prd-master` →「框架内置能力」** 已排除项（含本项目追加）生成单页 PRD。

### 总体与数据源

- 与 UI/API 根索引及明细文件同源（优先 `ui/`、`api/`）；**不**重复 `api/` 业务字段表全文；**不**编造规格未出现的字段名。

### 1. 基本信息

- **页面编号**：`PAGE-*`，与 `ui_manifest` **页面清单**、`ui/PAGE-*.md` 同源。
- **页面名称 / 所属终端**：终端名称与 `main_spec` **第4章终端说明** **逐字一致**。
- **所属版本**：与当前 **`versions/{版本号}/`** 一致。
- **操控形态**：**仅**「移动端」或「桌面端」，与 **`ui/PAGE-*.md` → 基本信息表** **一致**；**不**在基本信息表抄 L/D 骨架码。
- **MasterGo**：`ui_manifest` **`## 3`/`## 4`** 与本页同行；无则 **无**。
- **不写**：接口编号汇总表（见 **第 5 节**）。
- **与根规格关系（必填）**：紧接 **`## 1. 基本信息`** **表后**，落盘 **`> …` 引用块**（可多行简短列表，**不必**长篇）。必须让读者 **一眼**分清四份根文档分工，建议固定四件事：（1）布局分区：**`ui/PAGE-*`**；（2）接口语义：**`api/API-*`**；（3）功能规则：`feature_spec.md` / `feature/FEATURE-*`；（4）合法 `PAGE-*` 与跳转全集：**`ui_manifest.md` §3** / `ui/`。**禁止**在该引用块粘贴本 SKILL「产出具约束」全文或教学方法论。

### 2. 用户故事（本页）

单页 **第 2 节** 落盘 **仅** `## 2. 用户故事（本页）` + **`US-*` 表**。

- **与 `main_spec.md` → 1.2 的区分**：**1.2** 为 **产品级** 用户故事表；本节 **仅** 本页界面内可验证的「我要 / 以便」，**禁止**整表或无关行迁入。可 **对照 1.2** 裁剪角色与价值，并结合 **`ui/PAGE-*.md`** 的目标与任务。
- **与 1.2 的关系**：价值 **不与 1.2 矛盾**；**粒度、`US-*`、措辞** 与 **1.2** **区分**，**勿**沿用 1.2 行号或表结构。
- **与第 3 节**：条数按页裁剪；**`US-*`** 可被第 3 节引用（如「见 US-1」）。
- **补缺（可选）**：当 **`ui/PAGE-*.md`** 对本页 **可操作路径或验收粒度** 偏粗时，允许加厚本表。

### 3. 页面模块详述

- **整页线框（ASCII，默认必填）**：在 **`## 3. 页面模块详述`** 标题下、**第一个 `###` 分区之前**，用 **代码围栏**（`~~~` 或 **```** 均可）写 **自上而下** 的整页线框（`+--` / `┌` 等随意，**须可读**）。**须**与 **`ui/PAGE-*.md` → §2 页面层级** 的 **分区顺序、上下叠放、主辅区关系** **一致**，供下游 **AI / 代码生成** 对齐 **布局与路由挂载点**。仅当本页 **无实质 UI**（如纯重定向占位）时，围栏内写 **一行** `无 UI 线框` 即可。
- 顺序与分区：严格按 **`ui/PAGE-*.md` → §2** 自上而下；每个 PAGE 分区对应本文件一个 **`### {分区标题}`**，标题与 PAGE 明细内 **`####` / `#####`** **逐字对齐**（**勿**自拟另一套分区名）。
- 每分区固定三 bullet：**结构与控件**、**交互与跳转**、**涉及接口**（接口 **只** 写 **`API-*` 编号**，**不**写请求/响应字段）。
- **局部自定义UI组件分区**：**结构与控件** / **交互与跳转** **禁止**复述 `ui/COMP-*` 中的 **UI 状态矩阵**（loading/empty/error/edit 等）；首次出现处 **一行加粗** `**{组件名}（详见 ui/COMP-*.md）**` 即可。
- **加载 / 空态 / 异常 / 动效（页面级）**：仅写 **非组件内部** 的页面级编排（如首屏并行请求、路由级 overlay、与壳层并行刷新）；**组件分区** 的状态 **以 COMP 为准**。
- **补缺（可选，非全面加厚）**：当 **`ui/PAGE-*.md`** 在该 **`####`** 条目下 **未写清**与本页交付相关的 **页面级** 加载编排 / 弱网兜底 / 静默请求等（**非** COMP 内部态），允许在 **本条「交互与跳转」**内 **补几句**。**仍禁止**：①抄写 **`api/API-*.md` 业务字段表**或粘贴 JSON；②新增或与 **PAGE 明细层级顺序相悖**的自拟 **`###` 分区标题**；③把本节写成 **复述 COMP** 状态矩阵或组件长文。
- **出页**：在 **交互与跳转** 写 **控件级** 触发；**全页汇总** 在 **第 4 节**，与本节 **不矛盾**。
- **桌面端**：**不写**壳侧栏、壳顶栏等 **壳层换页**（与 **第 4 节** 桌面专项一致）。
- **分区内线框（可选）**：结构复杂的 **`###`** 下，可在三 bullet **之前** 再嵌 **小号** ASCII 块，细化该分区；**勿**与整页线框 **矛盾**。
- **局部自定义UI组件（若有）**：首次出现处 **一行加粗** `**{组件名}（详见 ui/COMP-*.md）**`；名称与 **`ui_manifest` §4**、`ui/COMP-*.md` **逐字一致**。

### 4. 页面跳转清单

- **表列（固定）**：**目标页面编号** | **目标页面名称** | **存放路径** | **触发条件** | **跳转类型** | **备注**（后四列含义同模板；**存放路径** 见下条）。
- **存放路径**：取 `ui/PAGE-*.md`（优先）或 `ui_manifest.md` 页面清单中该目标页记录的路径信息，**逐字一致**；若无则写 **无**。
- 目标页须在 `ui_manifest`/`ui/` 已登记 **PAGE-***；**禁止**未立项页。
- 与 **第 3 节**「交互与跳转」**同源核对**；表为 **全页出口**。
- **桌面端**：**仅**主工作区 / 页面内容区控件触发的出页；**不**收录业务壳（顶栏、侧栏、面包屑、壳层 Tab、右侧规格抽屉等）换页；壳由 **`lny-prd-ui/SKILL.md` →「桌面业务壳 PRD（最小结构）」** 约定落盘（如 `pages_prd/_shell/{终端编码}-shell.md`），**勿**在本单页罗列壳跳转。
- **跳转类型**：填该终端/技术栈常用表述（如小程序 `navigateTo`、Web `push`、Tab、外链等）。

### 5. API 交互清单

- **仅**三列索引：接口编号、本页用途一句、触发时机；**不**列请求/响应字段、**不**贴 JSON。
- 与 **第 3 节**「涉及接口」汇总 **一致**；**不**增删未在 `api_spec` 登记的编号。
- 同一接口多处触发：在「触发时机」用 **分号** 并列；过长可改为两列（**接口编号** | **用途与触发**）。

### 6. Feature 关联清单

- **仅**四列索引：Feature 编号、Feature 名称、本页关联点、来源路径。
- 数据来源为 `feature_spec.md` 与 `feature/FEATURE-*.md`；只收录与本页直接相关的 `FEATURE-*`。
- 若本页暂无关联 Feature，须写 **无**（不可留空）。
- 与第 3 节交互说明、第 5 节 API 清单保持一致（同一业务链路可互相追溯）。

## 版本与变更记录纪律

- 不得在根规范「变更记录」表新增行（v1.0.0 仅 `/lny-prd-master`；新版本行仅 `/lny-prd-iter`）
- 若本次修改了根目录四规范任一，须在 `versions/{版本号}/iteration_notes.md` 文末追加总结（见 `lny-prd-master` §1.1）；仅生成/更新 `pages_prd/` 下 PRD 且未改根规格者可省略
- 单页 PRD 含版本元信息时与当前 `versions/{版本号}/` 一致

## 前置条件
- 必须已执行 `/lny-prd-master` 初始化项目
- 必须已执行 `/lny-prd-ui` 完成用户界面设计
- 必须已执行 `/lny-prd-api` 完成 **`api_spec` 接口需求**（§4 索引 + `api/` 明细）
- `ui_manifest.md` 或 `ui/PAGE-*.md` 中必须包含该页面编号的定义
- 若由 **⑧** 委派进入：须存在 `versions/{版本号}/ui_changes.md`，且 **版本号与 iter 新建版本一致**

## 输入变量

```yaml
版本号: v1.0.0

接口编号列表: [API-MP-001, API-MP-002]   # 须为业务 API；登录等框架通用接口勿列入
MasterGo链接: (可选)
```

## 执行步骤
1. 检查 `版本号`、`页面编号`、`versions/{版本号}/` 是否存在
2. 若存在 `versions/{版本号}/ui_changes.md`：Read 本页面编号对应行，核对 `pages_prd目标路径`（规则见步骤 9）；不一致时以台账为准
3. 读取 `main_spec.md`（仅用于 §1.2 用户故事背景）+ `ui_manifest.md`/`ui/`（获取本页与跳转目标的合法 PAGE-* 及存放路径）
4. 读取接口需求（`api_spec.md` §4 + `api/API-*.md`）、UI 结构（`ui/PAGE-*.md` + 引用的 `ui/COMP-*.md`）、`feature_spec.md` Feature 约束
5. 读取本文「single_page_prd.md 模板」与「单页 PRD 产出具约束」全文后再落盘
6. 生成单页 PRD：骨架与模板节名/表头/线框围栏一致，取值增删悉遵约束。禁止在 `pages_prd` 写约束说明或重复一级章节。整页 ASCII 线框默认必填（无 UI 页例外）。
   - §1：基本信息表下一行 `> …` 根规格依赖（必填）
   - §2/§3：按约束补缺规则加厚根规格薄弱环节，不做无章法堆砌
   - §1～§6：依次对应约束 1.～6.
7. 保存 PRD：`pages/index` → `versions/{v}/pages_prd/pages/index/PAGE-MP-*.md`；`subpackages/{m}` → `versions/{v}/pages_prd/subpackages/{m}/PAGE-MP-*.md`；`views/{m}` → `versions/{v}/pages_prd/views/{m}/PAGE-AD-*.md`
8. 输出文件路径；若来自台账提示更新状态；若原型不存在提示可经总控 ⑥ 生成
9. 若修改了根目录四规范任一：在 `versions/{版本号}/iteration_notes.md` 文末追加总结；未改则省略

### 版本路径校验

执行前须存在 `versions/{版本号}/`；可与 `versions/` 目录下列出的文件夹名对照。未指定 `版本号` 时按 `lny-prd-master` **§1.1「当前工作版本号」** 取 `versions/` 下最大版本。
