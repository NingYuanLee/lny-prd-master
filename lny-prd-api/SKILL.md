## 与总控的关系

全流程见 **`lny-prd-master/SKILL.md`**。本步为 **③ `/lny-prd-api`**：维护 **`api_spec.md` 接口需求**（索引 + 规则）及 `api/` 明细。完成后总控通常进入 **④ `/lny-prd-feature`**，或 **⑤ `/lny-prd-page`** / **⑥ `/lny-prd-prototype`**（视用户意图）。

# 梳理接口需求 `/lny-prd-api`

> ⚠️ 注意：本步骤为第三步，需在完成第一步 `/lny-prd-master` 和第二步 **`/lny-prd-ui`**（用户界面设计，含页面架构）后执行。

## 总控入口（无需用户自选子指令）

若由 **总控**（**`@lny-prd-master` / `/lny-prd-master`** 等，见 `lny-prd-master`）进入：本步已判定须补接口需求；**完整 Read 本文后再执行**。用户回复「继续」即可由总控进入下一步，不必手动发 `/lny-prd-api`。

## 本步产出定位（API 接口需求）

本技能维护的 **`api_spec.md` + `api/`** 是 **产品 PRD 中的接口需求层**，描述「要什么能力、连哪些页、关键业务字段与规则」，**不是** OpenAPI / 实现规范。

- **开笔前**：Read **`lny-prd-master/SKILL.md` →「框架内置能力（不纳入 PRD）」**（含 **排除口径** 与「仍须写 PRD 的边界」）及本项目追加排除项。**不为** **框架通用** 已排除项创建 `api/API-*` / `api/EXT-*`；**业务** API/EXT **仍须**按 master 边界建档。
- **PRD 边界**：鉴权、域名、HTTP 方法、path、JSON 结构、响应 code 等 **属框架与后端实现**，**不在本技能展开**；§1 可写 **「由公司统一框架鉴权」** 一句。
- **禁止落盘 meta 文案**：`待补充`、`由后端Agent校准更新`、`草案`、`占位说明` 等。

## 目录化结构（索引 + 明细）

- 根索引：`api_spec.md`（阅读说明、响应形态约定、第三方清单、API 索引）
- 明细目录：`api/API-*.md`（单接口需求）、`api/EXT-*.md`（第三方业务需求）

执行时遵循：

1. **优先新结构**：维护 `api_spec.md` 索引 + `api/API-*.md` / `api/EXT-*.md` 明细；**以 `api/` 为明细单一事实源**。
2. **兼容旧结构**：若仍为旧版 `api_spec.md` §5 单体大段，**提示迁移**至 `api/API-*.md` 后改维护索引模式。

## 写作规范

### 1. 终端与索引（§4）

| 规则 | 说明 |
|------|------|
| 终端来源 | 仅按 `main_spec.md` → 第4章终端说明 划分小节，不得臆造或遗漏 |
| 索引列 | 序号、接口编号、描述、涉及页面、明细路径 |
| 禁止 | 服务基地址、请求路由 |

### 2. 第三方接口（§2）

- 维护 §2 清单（索引 \| EXT编号 \| 名称/用途 \| 服务商 \| 明细路径）；无第三方写 **无**
- `EXT-{三位序号}`，与 `api/EXT-*.md` 一一对应；业务细则写明细（产品语言，不写协议/签名/回调 URL）

### 3. 响应形态（§3，含 §1 默认写"由公司统一框架鉴权"）

| 形态 | 含义 | 典型场景 |
|------|------|----------|
| **基础对象** | 返回单个业务对象或操作结果 | 详情、提交、状态变更 |
| **无分页列表** | 返回完整列表（数据量可控） | 下拉选项、短列表 |
| **分页列表** | 返回列表 + 分页信息 | 列表页、搜索结果 |

分页列表类：请求须含当前页码、每页条数；响应须含列表数据、总条数。单 API 明细「响应形态」须与 §3 三类之一一致。

### 4. 参数字段表（`api/API-*.md`）

| 业务字段 | 类型语义 | 必要 | 说明 |
|----------|----------|:--:|------|

- 业务字段：中文名（与 UI 一致），不写 snake_case/JSON 路径
- 类型语义：字符串/数值/布尔/数组/对象/日期时间等，与 UI 一致（多选→数组，开关→布尔）
- 说明：枚举值、默认值、单位、校验口径；无则 **—**
- 配置类接口：须写明读写粒度，禁止模糊"配置表"

### 5. 单 API 明细必备内容

接口用途、触发与场景、请求/响应需求（字段表）、业务规则与异常、关联 Feature 清单（无则写 **无**）、页面固定参数（无则 **无**）。

## 立项落盘与 `api/` 目录初始化

**`/lny-prd-master` 立项时**：创建空目录 **`api/`**，按上文 **「api_spec.md 模板」** 生成根目录 **`api_spec.md`**（索引骨架）。

**`/lny-prd-api` 续跑时**：

1. 若 **`api_spec.md` 不存在**：按 **「api_spec.md 模板」** 在根目录创建；同步创建 **`api/`**（若不存在）。
2. 若 **`api/` 不存在**：创建空目录；旧版单体 `api_spec.md` §5 大段须 **迁移** 至 `api/API-*.md` 后改维护索引模式。
3. 新增 **`API-*`**：在 **`api_spec.md` §4** 增索引行 → 按 **「单 API 模板」** 创建 **`api/API-{终端}-{序号}.md`** → 更新 **`main_spec.md` §6** 有效 API 数（若项目维护该统计）。
4. 新增 **`EXT-*`**：在 **§2** 增清单行 → 按 **「单 EXT 模板」** 创建 **`api/EXT-{序号}.md`**。
5. **禁止**在 `api/` 存放原型、UI、Feature 正文。

## 前置条件

## 功能

根据 `ui_manifest.md` / `ui/` 的页面索引明细，为 **`api_spec.md`**（索引）与 `api/API-*.md`（明细）建立 **接口需求**；并为 **`api_spec.md` §2** 建立 `api/EXT-*.md` 明细。旧版项目 **提示迁移** 单体 §5 至 `api/`。

## 输入方式

仅支持 **一次性输入** 与 **Agent 对话式补充**（普通聊天消息）。

### 一次性输入（推荐）

```yaml
版本号: v1.0.0
操作模式: add 或 modify
接口信息:
  - 接口编号: API-MP-001   # 修改时必填，新增时不填
    接口名称: 获取商品列表
    关联页面: [PAGE-MP-002, PAGE-MP-003]
    接口用途: 获取商品列表数据
    响应形态: 分页列表
    触发场景: 进入商品列表页时加载
    请求参数:
      - { 业务字段: 当前页码, 类型语义: 数值, 必要: 是 }
      - { 业务字段: 每页条数, 类型语义: 数值, 必要: 是 }
      - { 业务字段: 排序方式, 类型语义: 字符串, 必要: 否, 说明: "枚举：综合/价升/价降/销量" }
    响应参数:
      - { 业务字段: 列表数据, 类型语义: 数组, 必要: 是 }
      - { 业务字段: 总条数, 类型语义: 数值, 必要: 是 }
第三方接口:
  - { 接口编号: EXT-001, 接口名称: 微信统一下单, 服务商: 微信支付, 关联页面: [PAGE-MP-005], 关联功能: [FEATURE-003] }
```

### 对话式补充

若用户仅发送 `/lny-prd-api` 或未带齐字段：在对话中用自然语言列出缺失项，请用户补全后再落盘。

### 版本路径校验

使用某 `版本号` 前须存在 `versions/{版本号}/`；不存在则提示用户更正或先完成立项/迭代。

## 执行步骤

1. 检查 `main_spec.md`、`api_spec.md`（不存在则按模板创建 + 建 `api/` 目录）
2. 读取 `ui_manifest.md`/`ui/` 获取页面列表，读取 `main_spec.md` 第4章校验终端
3. 读取 `api_spec.md` 与 `api/` 明细（若存在）
4. 校验页面关联：页面须在 `ui/`/`ui_manifest` 中存在；终端编码与第4章一致
5. 梳理接口并核对 `feature_spec.md` 中相关 `FEATURE-*`
6. 目录化模式：更新 `api_spec.md` 索引+§1～§5 → 新增/修改 `api/API-*.md`、`api/EXT-*.md`；旧版兼容：存在单体 §5 大段则提示迁移至 `api/`
7. 更新 `versions/{版本号}/iteration_notes.md` 文末总结
