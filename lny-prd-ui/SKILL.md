## 目录化兼容模式（索引 + 明细）

为避免 `ui_manifest.md` 持续膨胀，本技能支持并推荐：

- 根索引：`ui_manifest.md`（目录、规则、页面/组件索引）
- 明细目录：`ui/PAGE-*.md`、`ui/COMP-*.md`

执行时遵循：

1. **默认结构（推荐）**：`ui_manifest.md` **仅**维护 **§1 约定、§2 框架、§3 页面索引、§4 组件索引、§5 全局特殊说明**；**所有**页面/组件详述写在 **`ui/PAGE-*.md`、`ui/COMP-*.md`**，与索引表「明细路径」列 **一一对应**。
2. **禁止双写**：**不得**在 `ui_manifest.md` 内再设「分页面详述」「组件详述」等大段正文（历史 **`## 5` 组件节 / `## 6` 页面节** 须 **迁移** 至 `ui/` 后 **删除**）。
3. **旧项目迁移**：若 manifest 仍含全量页面/组件正文，**优先**将正文下沉至 `ui/` 后 manifest **只留索引**（可参考项目内 `scripts/migrate-prd-structure.mjs` 脚本）；迁移完成前 **不得**继续双轨扩写。

## `ui_manifest.md` 产物规范

**分工**：`ui_manifest.md` = **索引 + 全局约定**；`ui/PAGE-*.md`、`ui/COMP-*.md` = **详述单一事实源**。模板 **§1** 面向读者与写作者（层级 L0～L5、D0～D5、控件形态）；**§1.5** 约定 **`ui/` 明细体例**（见本文 **「ui_manifest.md 模板」** 与下文 PAGE/COMP 模板）。下列 **第 4、6 点** 中的布局/版式规则 **落盘目标为 `ui/PAGE-*.md` / `ui/COMP-*.md`**，**不是** manifest 正文。

1. **不写「主体风格和色调」**：本清单不承担品牌色、主题气质等描述（已从模板移除）。
2. **UI 框架（按终端表格）**：依据 **`main_spec.md` → 第4章终端说明** 逐行建表（**只读**，本步 **不得** 改第4章）；列明终端、类型编码、**UI 框架名称**、**官方文档或主页 URL**。已知则填实值，未知则在对话中向用户确认后落盘 **无**，**禁止编造链接**，**禁止**写 `待补充`。若 `ui_manifest` 所需终端未在第4章列出，**停止落盘**并提示用户先走 **`/lny-prd-master`** 新增终端。
3. **操控形态定义（全文区分基准）**：
   - **移动端**：以 **手指触控** 为主交互方式（手机/平板等便携触控设备上的小程序、App、H5 等，以项目实际终端为准）。
   - **桌面端**：以 **鼠标指针**（及键盘辅助）为主交互方式（PC 网页、管理后台 Web 等）。
   - **书写顺序**：全文先明确各终端/页面归属 **移动端** 或 **桌面端**，再按对应小节撰写 **层级及布局、分页面描述**；同一项目多端并存时 **分别叙述**，不得默认「一种交互套所有端」。
4. **层级及布局规范（按操控形态分叉）**：
   - **通用原则**（两端均适用）：**自底至顶（层级优先）** → **自上而下（同层级不同模块）** → **自左至右（左右布局优先，同模块内子项）** → **自外至内（同一父子关系由内向外展开）**。
   - **移动端**：沿用模板 **§1.3** 的 **层级一览表（L0～L5）** 与分页面展开方式；布局上 **尽量采用自动弹性容器**（弹性盒、自适应流式/栅格等），减少写死像素分区。
   - **桌面端**：在通用原则之上，必须体现模板 **§1.4** 的 **布局结构类型** 与 **内容区固定结构**：
     - **空白布局**：独立全页、无业务壳（**框架内置鉴权类全页** 见 **`lny-prd-master` →「框架内置能力」**，通常不建 PAGE）。
     - **基本布局（概念）**：**D1-1 / D1-2 / D1-3** 均须 **顶栏 + 侧栏 + 内容区** 壳；**面包屑** 属壳层，须在 **`ui/PAGE-*.md`** 写清。**骨架码只写 D1-*，勿写 `D1`**。
     - **标准内容区（D1-1）**（自上而下固定顺序）：**搜索栏** → **功能栏** → **列表区**（须含 **操作列**）→ **汇总 + 分页栏**。**（1）～（4）** 须在对应 **`ui/PAGE-*.md`** 写清并与 **`api/API-*.md` 请求/响应需求** 同源；**禁止**虚构字段。
       - **（1）搜索栏**：**栅格化**排布搜索条件；每一项条件须 **对齐** 所引用查询接口 **请求需求** 中的 **业务字段**，**不得**增加虚构字段。
      - **（2）功能栏与操作列**：**功能栏** 与 **列表操作列** 同属 **界面功能**；须与 **`ui_manifest` / PRD 功能点描述** 对齐；该页须附 **页面功能清单**（表或列表），逐条标明功能归属 **功能栏** 或 **操作列**、**限制条件**（权限、状态、批量/单行等）；每个功能按钮须写明 **打开的浮层类型** 或 **触发的反馈**（与 **（4）** / **§1.4.1 D5** 及 **关联 API 需求** 一致）。
       - **（3）列表区**：选用合适 **表格** 形态；**展示列**（非操作列）须 **对齐** 列表/详情等接口 **响应需求** 中的 **业务字段**，**不得**展示不存在的字段。
       - **（4）浮层与反馈（§1.4.1 D5）**：弹窗（表单 / 列表 / 展示）、反馈链等须 **对齐 `API-*`**；**勿**与 **§1.3.1 L5** 混码。
     - **表单内容区**（须区分两种状态）：**数据展示** / **数据录入**；操作区须体现 **编辑** 与 **保存 + 取消**（随状态切换显隐或可用性）。
     - **自定义内容区**：数据展示、特殊业务等非标结构，在分页面中逐项写清模块与交互。
     - **控件与表单形态（桌面端）**：在 **`ui/PAGE-*.md`** 中写清搜索区、内容表单、**表单弹窗** 所用控件（见下文分节）。
       - **搜索区表单**（别名 **筛选区**；含 **弹窗内** 承担搜索/筛选的表单）：
         - **字符串** → **输入框**。
         - **枚举**（单选或多选）→ **选择器**；**禁止**使用单选框组、多选框组。枚举项 **较多** 时，选择器须 **支持搜索**（可输入过滤选项）。
         - **2～3 个联动字段** → **级联选择器**；层级或选项 **较多** 时，级联选择器须 **支持搜索**（按框架能力实现）。
         - **起止日期或时间** → **时间范围选择器**（日期范围、时间范围或日期时间范围，按字段语义选用）。
       - **内容表单**（**表单内容区**）及 **弹窗表单**（**表单弹窗**）：
         - **字符串** → **输入框**。
         - **枚举**（单选或多选）→ **优先单选框组或多选框组**；**是否类**（布尔）→ **开关**。枚举项 **较多** 时，可 **改回选择器**（可带搜索）。
         - **2～3 个联动字段** → **多个选择器联动**（上一选择结果驱动下一项数据源）；**尽量不使用级联选择器**。
         - **起止日期或时间** → **时间选择器**（按起、止分别选）；**谨慎使用范围选择器**（仅在交互明确要求合并为一组控件时使用）。
   - **层级一览表**：移动 **L0～L5** 见 **`lny-prd-master` §1.3.1**；桌面 **D0 / D1-1～D1-3 / D5** 见 **§1.4.1**（**D5**≠**L5**）。可与 **L0～L5** 对照理解，不得省略 **§1.4** 结构约定。
5. **页面索引与分包（`ui_manifest` `## 3`）**：小程序项目须在 **`### 3.1`** 维护分包与 TabBar；**`### 3.2`** 维护页面索引表（含 **明细路径** → `ui/PAGE-*.md`；**MP 页面须含「小程序路由」列**）。`MasterGo` 无链接则写 **无**。
6. **层级及布局描述（按页，落盘 `ui/PAGE-*.md`）**：先标明 **移动端** 或 **桌面端** 与 **骨架码**（D0 / D1-1～D1-3 / L0～L5）；桌面 **D1-1** 须 **（1）～（4）**；**D5** 浮层另 **`#### │ D5`**。**版式**：页内 **`####` → `#####`**；可选 **U+2502 `│`** 前缀（仅标题行）；相邻逻辑块之间可用 **`---`**；列表用 `-`，子项行首两空格。**禁止**在 manifest 重复上述正文。
7. **局部自定义UI组件详述（落盘 `ui/COMP-*.md`）**：含 **设计前提**、**UI 状态矩阵**、结构与交互；**§4** 索引与 **`ui/COMP-*.md`** 双向一致；写 COMP 前须判定 **设计前提**（见下文）。
8. **特殊说明（`ui_manifest` `## 5`）**：仅 **跨页全局** 约定（如 Design Tokens、侧栏菜单结构）；**不得**在此写单页布局正文。若无则写 **无**。

## 局部自定义UI组件：判定、边界与状态职责

本节为 **`ui/COMP-*.md` 与 `ui/PAGE-*.md` 边界的单一事实源**；`lny-prd-master`、`lny-prd-page`、`lny-prd-prototype` 须引用本节，**不得**另立冲突规则。

### 术语

统一称 **「局部自定义UI组件」**；**索引**在 `ui_manifest.md` **§4**，**详述**在 **`ui/COMP-*.md`**。

### 抽组件判定（满足其一即可）

1. **多页复用**：已经或 **可能** 在多个页面重复出现。
2. **多态展示**：有多种状态展示形式（如 loading / empty / error / edit / view 及业务态）。

**不满足**上述任一条件时，UI 说明写在 **`ui/PAGE-*.md`** 对应分区内，**不**单独建 `COMP-*`。

### 聚合优先（反过度拆分）

局部组件 **并非越细越好**。**禁止**为追求「细粒度」而拆碎：

- 受 **同一状态机** 统一驱动、且 **视觉上相邻** 的区块 → **合并为一个 COMP**，即便视觉上几乎覆盖全页主内容区也 **不** 因「占面积大」而强行拆成多个 COMP。
- 是否拆 COMP 看 **状态边界 / 复用边界**，**不看** 面积大小。

若 COMP 由多块视觉分区组成，须在 `ui/COMP-*.md` 中说明 **为何同属一个 COMP**（同一状态机 + 视觉相邻），并写清 **状态机与区块联动**。

### PAGE 与 COMP 边界

| 载体 | 职责 |
|------|------|
| **`ui/PAGE-*.md`（整体）** | 层级布局、分区顺序、路由跳转、页面级浮层（L5/D5 中 **页面 orchestration** 部分）、局部组件 **引用表**；**纯组合**，不承载子组件内部状态详述 |
| **`ui/COMP-*.md`（局部）** | 结构、交互、数据接口、**全部 UI 状态**（含 loading/empty/error/edit 及业务态）；多区块时写清 **状态机 / 区块联动** |

**状态纪律（原子设计 + 单职责）**：

- **组件承载状态，页面纯组合**；页面只负责排版和路由。
- 每个局部自定义UI组件 **自行管理** 本组件范围内的 loading/empty/error/edit 等状态。
- **`ui/PAGE-*`** 中引用组件时：**一行加粗** `**{组件名}（详见 ui/COMP-*.md）**`，**禁止**重写状态矩阵或状态切换详述。
- **COMP §3 状态切换入口**：**仅**页内控件；**禁止**把预览壳 §C 写入 `ui/COMP`（§C 由 **`/lny-prd-prototype`** 按矩阵自动推断）。
- **L5 / D5（页面级）**：仅写 **页面级** 浮层（路由级全屏弹窗、页面 Toast 等）；**禁止**在此写属于子组件内部的状态切换详述。

### 拆页强制（复杂度下放）

当单页内 **≥3 个彼此独立的状态机 / 数据域**（各自需 loading/empty/error 等）时，**必须拆页**（水平切分 / 复杂度下放），宁可牺牲一点体验连贯性，也要保证 **每页、每组件极度简单**。

- **不**因单个 COMP 视觉面积大而拆页或拆 COMP。
- 拆页时在 `ui/PAGE-*.md` 填写 **「页面拆分说明」**（见 PAGE 模板）。

- **⑧ 委派进入**：Read `versions/{版本号}/ui_changes.md`，按台账中 **委派步=②** 且 **状态=待②**（或 `待②/待⑤`）的 **PAGE-*** 条目逐项执行；完成后在 `iteration_notes.md` **文末追加** 摘要（见 `lny-prd-master` **§1.1**）

## `ui_manifest.md` 初始化流程（续跑兜底）

若检测到 `ui_manifest.md` 不存在：

1. 按本文 **「ui_manifest.md 模板」** 在根目录创建
2. 按 **`main_spec.md` → 第4章终端说明** 预填 **§2 UI 框架** 表的终端行（框架名与 URL 未知时在对话确认后写 **无**）
3. 创建 **`ui/`** 空目录（若不存在）
4. 继续本步的页面清单与架构表补充

## 桌面业务壳 PRD（最小结构）

`versions/{版本号}/pages_prd/_shell/{终端类型编码}-shell.md`（如 `AD-shell.md`），**每桌面终端一份**，与 **`main_spec.md` → 第4章终端说明** 该端 **终端名称** 对应。

## 功能

为已立项的项目补充或修改 **页面与组件索引** 及 **UI 清单**。目录化项目优先维护 `ui_manifest.md` 索引与 `ui/` 明细；旧版项目可继续在 `ui_manifest.md` 全文维护。须在用户一次性输入或对话中明确 **版本号** 与操作意图，再逐个终端、逐个页面推进（可与用户自然语言穿插）。

## 输入方式

仅支持 **一次性输入** 与 **Agent 对话式补充**（普通聊天消息）。

### 一次性输入（推荐）
```yaml
版本号: v1.0.0
操作模式: add 或 modify
页面信息:
  - 页面编号: PAGE-MP-001   # 修改时必填，新增时不填
    页面名称: 首页          # 修改时不填，新增时结合语义自动
    所属模块: 首页          # 修改时不填，新增时结合语义自动
    主包分包: 主包
    功能模块: [搜索栏, 轮播图, 推荐列表, 底部Tab栏]
    页面跳转: [PAGE-MP-002, PAGE-MP-003]
```

### 对话式补充

若用户仅发送 `/lny-prd-ui` 或未带齐字段：在对话中用自然语言列出缺失项（版本号、`add`/`modify`、页面编号或终端等），可列出 `versions/` 与 `ui_manifest.md`/`ui/` 已有索引供对照；请用户在下一条消息用 YAML 或自然语言补全。落盘前可对单页输出「页面信息确认」摘要，由用户在对话中确认或修正。

### 版本路径校验

使用某 `版本号` 前须存在 `versions/{版本号}/` 且含 `iteration_notes.md`；不存在则在对话中提示用户更正版本号或先完成立项/迭代。

## 执行步骤

1. 检查 `main_spec.md` 是否存在
2. 检查 `ui_manifest.md` 是否存在；若不存在则按本文 **「ui_manifest.md 模板」** 在根目录创建，并创建 **`ui/`** 目录（立项应已落盘，本步为兜底）
3. 读取 `main_spec.md` **第4章**（只读）获取终端表，并读取 `ui_manifest.md`/`ui/` 获取现有页面与组件索引；若目标终端不在第4章，**停止**并提示走 **`/lny-prd-master`** 新增终端
4. **处理小程序分包配置**（项目含 **MP** 终端时）：
   - 记录分包模块划分、路径前缀、TabBar 说明
   - 确定主包页面（底部 Tab 页面）与各分包页面归属
   - 写入 `ui_manifest.md` **`## 3` → `### 3.1`**（非 MP 项目可省略 3.1）
5. 根据用户提供的页面信息处理：
   - **新增页面**：自动分配下一个可用页面编号，自动判断主包/分包
   - **修改页面**：根据指定页面编号找到对应行更新
6. 更新 `ui_manifest.md` **§3.2** 页面索引（及 `ui/PAGE-*.md` 必要字段），标注主包/分包；同步 **§4** 组件索引（若涉及）
7. 更新 `main_spec.md` **§5** 有效页面/组件统计（及 §6 / §7 等索引章节的同类统计表，若本次变更影响数量）；**同步** `ui_manifest.md` **§2 UI 框架** 表行与 **第4章终端说明** 一致（**只读**第4章，**不得** 修改第4章任何内容）
8. 目录化模式（存在 `ui/` 时）：更新 `ui_manifest.md` 索引与规则，并写入 `ui/PAGE-*.md`、`ui/COMP-*.md` 明细
9. 旧版兼容模式（无 `ui/` 时）：更新 `ui_manifest.md` 全文（用户端页面清单、阅读说明、分页面、局部自定义UI组件、特殊说明）
10. 若涉及局部自定义UI组件：同步更新 `ui_manifest.md` 与 `ui/COMP-*.md`（若存在）对应内容；新建 COMP 须含 **设计前提** 与 **UI 状态矩阵**
11. 更新 **`versions/{版本号}/iteration_notes.md`**：在 **文末**（或文件内已有 **「变更流水」** 等同级小节之末）**按时间顺序追加** 本次对 **`main_spec.md`、`ui_manifest.md`**（及 `ui/` 若存在）的 **总结**；**不得** 在根目录规范 **「变更记录」** 表中新增行；**不得** 新建 `versions/` 下更高版本号目录或虚构新版本号。若文件尚无流水小节，可新起 `## 变更流水` 后再逐条追加。
12. 输出修改的文件列表和页面编号分配结果
