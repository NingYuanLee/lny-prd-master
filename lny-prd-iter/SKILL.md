## 与总控的关系

全流程见 **`lny-prd-master/SKILL.md`**。本步为 **⑧ `/lny-prd-iter`**。本步 **只** 建版本壳、写变更台账、登记根规范「变更记录」一行，并输出 **委派清单**；规格与单页 PRD 正文由 **②③④⑤** 按台账执行。完成后 **控制权交还总控**：用户回复「继续」触发 master §3 I2，总控按委派清单依次驱动 **②→③→④→⑤→⑥→⑦**（每轮一个步，按需跳过无待办项）。

# 新迭代 `/lny-prd-iter`

## 角色边界

| 本步只做 | 本步禁止 |
|----------|----------|
| 版本号校验，创建 `versions/{新版本号}/` | 写入/修改 `ui/PAGE-*.md`、`api/API-*.md`、`feature/FEATURE-*.md` 正文 |
| 写 `iteration_notes.md`（开篇：目标/统计/委派总览） | 生成 `versions/{v}/pages_prd/**/*.md` 内容 |
| 写三类台账：`ui_changes.md`、`api_changes.md`、`feature_changes.md` | 更新 `main_spec` 页面架构表、`api_spec` §4 索引、`ui_manifest` §2-§4、`feature_spec` 索引行（留给 ②③④） |
| 新增/修改页面在 `ui_changes.md` 登记 `pages_prd` 目标路径，状态初始 `待⑤` | 在输出中声称「已生成单页 PRD」「已登记 feature_spec」等子技能动作 |
| 在根目录四规范「变更记录」表各追加一行（仅版本号+简述+日期；不改 §5/§6/§7） | |
| 输出委派清单（建议顺序：②→③→④→⑤→⑥→⑦） | |

## 规范参考

- 版本格式：`v{主}.{次}.{修订}`（如 v1.0.0、v1.1.0）
- 版本文件夹：`versions/v{主}.{次}.{修订}/`
- 核心规格文件始终为最新状态；废弃内容由 **②③④** 在索引中标记 `deprecated`，保留明细文件
- 版本文件夹存放当次迭代增量：`iteration_notes.md`、`*_changes.md` 台账；`pages_prd/` 由 **⑤** 落盘

## 版本与变更记录（本技能独占）

- **`v1.0.0` 的初始化**（`versions/v1.0.0/` 与各规范变更记录 **首行**）**仅** 由 **`/lny-prd-master` 立项** 完成；本技能 **不** 代为初始化 `v1.0.0`。
- **当且仅当**执行本技能 **`/lny-prd-iter`** 时，才允许：**新建** `versions/{新版本号}/`（新版本号 **大于** 当前最新）、生成 `iteration_notes.md` / `api_changes.md` / `ui_changes.md` / `feature_changes.md`，并在 **`main_spec.md`、`api_spec.md`、`ui_manifest.md`、`feature_spec.md`**（及项目模板中若另有根目录规范含「变更记录」表）中 **追加** 该新版本对应的 **一行** 变更记录。
- **禁止**：为同一版本号 **追加第二行**；在表中写出 **`versions/` 下不存在** 的版本文件夹名；或遗漏 **`versions/{新版本号}/` 已创建** 却未在应维护的各规范表中登记的情况。
- 迭代完成后：**「变更记录」表中的版本列表** 与 **`versions/`** 目录下子文件夹名 **一一对应**。
- **`iteration_notes.md` 与过程性流水**：由 **⑧** 写 **开篇**；**②、③、④、⑤** 在 **`versions/{版本号}/iteration_notes.md`** **文末按序追加** 过程性流水（详见 **`lny-prd-master/SKILL.md` §1.1 + §1.1.1** 过滤规则）；**⑥、⑦** **不**维护该流水。

## 九类变更

| 序号 | 制品 | 操作 |
|:----:|------|------|
| 1 | 页面 | 新增 |
| 2 | 页面 | 修改 |
| 3 | 页面 | 废弃 |
| 4 | 接口 | 新增 |
| 5 | 接口 | 修改 |
| 6 | 接口 | 废弃 |
| 7 | 功能 | 新增 |
| 8 | 功能 | 修改 |
| 9 | 功能 | 废弃 |

## 四类委派映射

| 委派步 | 触发条件 | 台账文件 | 备注 |
|:-----:|---------|---------|------|
| ② | 页面有新增/修改/废弃 | `ui_changes.md` | 废弃 = 索引 `deprecated`，保留 `ui/` 文件 |
| ③ | 接口有新增/修改/废弃 | `api_changes.md` | 废弃 = 索引标记，保留 `api/` 明细 |
| ④ | 功能有新增/修改/废弃 | `feature_changes.md` | |
| ⑤ | 任一变更在页面上有**可感知变化**（页面增删改、新增接口的字段/按钮/埋点、功能在页面的落地等） | `ui_changes.md` 的 `pages_prd目标路径` 列 | 路径规则对齐 `lny-prd-page` 执行步骤 9 |

同一页面若 **新增** 或 **修改**：台账 **一行** 覆盖 UI（委派 ②）与单页 PRD（委派 ⑤）；`pages_prd目标路径` 必填，`状态` 对 ② 为 `待②`、对 ⑤ 可另记或合并为 `待②/待⑤`（见台账说明）。

## 自然语言解析

用户可给结构化 YAML，也可给自然语言；本步须将输入解析为 **0～N 条** 台账行，每条映射九类变更之一：

1. **拆分**：一句意图可产生多行（例：「首页改版并加登录接口」→ `修改页面 PAGE-MP-001` + `新增接口 API-MP-0xx`）
2. **ID 已知**：直接填入 `ID` 列（`PAGE-*` / `API-*` / `FEATURE-*`）
3. **ID 未知**：`ID` 写 `待确认`，`摘要` 写用户原意，`委派步` 仍标明（如疑似修改页面 → `②`）
4. **废弃**：关键词「下线/废弃/删除/不再使用」且对象为页面/接口/功能时，操作列填 **废弃**（不删文件，由子技能改索引）
5. **勿**再使用模糊的 `UI变更: 描述文字`；UI 相关须落到 **新增/修改/废弃页面** 之一

## 前置条件

- 必须已执行 `/lny-prd-master` 初始化项目
- `main_spec.md` 中必须有变更记录表

## 输入变量

```yaml
版本号: v1.1.0
迭代内容简述: 一句话
新增页面: [PAGE-MP-004]
修改页面: [PAGE-MP-001]
废弃页面: [PAGE-MP-003]
新增接口: [API-MP-010]
修改接口: [API-MP-002]
废弃接口: [API-MP-003]
新增功能: [FEATURE-010]
修改功能: [FEATURE-003]
废弃功能: [FEATURE-001]
```

说明：功能编号按跨端规则 `FEATURE-{三位序号}`；历史 `FEATURE-MP-001` 等旧编号可继续出现在迭代输入中。列表项可省略（表示本类无变更）。

## 统一台账表结构

三类 `*_changes.md` 均采用下列 Markdown 表（无变更时保留表头，正文写 **无** 或空表）：

```markdown
# {UI|API|Feature} 变更台账 - {版本号}

| ID | 操作 | 摘要 | 委派步 | pages_prd目标路径 | 状态 |
|----|------|------|--------|-------------------|------|
| PAGE-MP-004 | 新增 | {一句话} | ② | versions/{v}/pages_prd/pages/index/PAGE-MP-004.md | 待② |
| PAGE-MP-001 | 修改 | {一句话} | ② | versions/{v}/pages_prd/pages/index/PAGE-MP-001.md | 待② |
| PAGE-MP-003 | 废弃 | {一句话} | ② | 无需单页 PRD | 待② |
| API-MP-010 | 新增 | {一句话} | ③ | — | 待③ |
```

- **状态枚举**：`待②` / `待③` / `待④` / `待⑤` / `已完成`（本步 **只初始化**；子技能完成后可在 `iteration_notes` 流水注明，或人工改台账）
- **pages_prd目标路径**：**仅**「新增/修改页面」填写；规则见 `lny-prd-page` 步骤 9（从 `ui/PAGE-*.md` 或 `ui_manifest` 的 **存放路径** 推导 `versions/{版本号}/pages_prd/{路径}/PAGE-*.md`）；**废弃页面**填 `无需单页 PRD` 或 `—`
- **api_changes.md / feature_changes.md**：`pages_prd目标路径` 列统一填 `—`

### `iteration_notes.md` 开篇结构（最小）

```markdown
# 迭代摘要 - {版本号}

## 版本目标
{迭代内容简述}

## 变更统计
- 页面：新增 n / 修改 n / 废弃 n
- 接口：新增 n / 修改 n / 废弃 n
- 功能：新增 n / 修改 n / 废弃 n

## 委派顺序
1. `/lny-prd-ui`（②）— {待办 PAGE 摘要}
2. `/lny-prd-api`（③）— {待办 API 摘要}
3. `/lny-prd-feature`（④）— {待办 FEATURE 摘要}
4. `/lny-prd-page`（⑤）— {待办 pages_prd 路径列表}
5. `/lny-prd-prototype`（⑥）— 有新增/修改页面时必做；仅 API/Feature 变更时可选
6. `/lny-prd-check`（⑦）— 建议
```

### `iteration_notes.md` 流水记录速查

> 写入规则全文见 **`lny-prd-master/SKILL.md` §1.1.1 / §1.1.2**（流水内容过滤规则 + 自检清单）。此处不重复。

## 版本号校验

1. 检查格式是否符合 `v{主}.{次}.{修订}`
2. 检查是否 **>** 当前最新版本号（以 **`versions/`** 目录为准）
3. 语义化版本合理性校验：
   - 合理递增：v1.2.3 → v1.2.4、v1.3.0、v2.0.0
   - 跳跃递增：允许但警告「版本跳跃较大，请确认」
   - 递减：不允许，直接报错

## 执行步骤

1. 校验 `版本号` 参数
2. 以 **`versions/`** 目录下已有文件夹名为 **准**，与 `main_spec.md`（及他规范）变更记录交叉核对，确定 **当前最新版本号**；二者冲突时 **以 `versions/` 为准** 并在对话中说明
3. **校验版本号**：不大于当前版本号则 ❌ 失败；大于则继续
4. **解析输入**：YAML 或自然语言 → 九类台账条目（见「自然语言解析」）
5. 创建 `versions/{新版本号}/` 文件夹
6. 生成 `iteration_notes.md`（开篇结构 + 委派总览）
7. 生成 `ui_changes.md`、`api_changes.md`、`feature_changes.md`（统一表结构；为新增/修改页面推导 `pages_prd目标路径`）
8. 在 `main_spec.md`、`api_spec.md`、`ui_manifest.md`、`feature_spec.md` 中：
   - **「变更记录」表各追加一行**（版本号、简述、日期）
   - **「文档版本」字段**更新为新版本号，**「最后更新」日期**更新为当天
   - **不**改索引（如 §3/§4/§4.2）、统计（§5/§6/§7）、架构表正文
9. 输出委派清单 + 所有新建/修改文件路径；**不**声称已写规格或单页 PRD
10. **自检提示**（输出中列出）：所有「新增/修改页面」的 `pages_prd目标路径`，供 **⑤** 落盘与 **⑦** 核对

## 输入与交互约定

- **一次性输入**：触发 `/lny-prd-iter` 时在同一条消息中给出「输入变量」或自然语言描述（新版本号、迭代内容、增删改列表等）。
- **对话式补充**：版本号不合法或与 `versions/` 现状冲突时，在普通 Agent 对话中说明并请用户下一条消息修正。

### 版本路径与递增

新版本号须大于 `versions/` 下当前最新文件夹名所表示的版本；创建前在对话中可与用户对齐命名与递增规则。
