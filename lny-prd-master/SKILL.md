### 1.1 版本号与变更记录（全局纪律）

- `**v1.0.0` 与 `versions/v1.0.0/`**  
**当且仅当**执行本文 **「立项 `/lny-prd-master`」** 并在空目录完成脚手架生成时，才创建 `versions/v1.0.0/`，并将各根目录规范（如 `main_spec.md`、`api_spec.md`、`ui_manifest.md` 等）中的 **「文档版本」** 与 **「变更记录」** 表写为 **首版 v1.0.0**（每文档该版本 **一行**）。  
**总控续跑**（已存在 `main_spec.md`）及 **②～④、⑤～⑧** 各子技能执行时：**不得**新建 `versions/v1.0.0/`、**不得**在变更记录中再增 **v1.0.0** 行或产生 **同版本重复行**。（**⑧** 不得用于「补初始化」`v1.0.0`，仅用于高于当前最新的新版本；见 `lny-prd-iter`。）
- **新版本与变更记录追加**  
**当且仅当**执行 `**/lny-prd-iter`**（并 Read `lny-prd-iter/SKILL.md`）时，才允许 **新建** `versions/{新版本号}/` 并在各规范 **「变更记录」** 表中 **追加** 对应新版本行（具体内容与范围以 `lny-prd-iter` 为准）。
- **其余技能（含总控代跑的 ②～④、⑤～⑧）**  
**不得**在任何规范文档的 **「变更记录」** 表中 **新增行**（不得以「记一笔」方式反映当次编辑）。  
**同一版本号**在变更记录表中 **不得**出现 **多行**；若发现重复，须在后续 `**/lny-prd-iter`** 或人工整理中合并为每版本一行（其他技能 **不**擅自改表，可交由检查步报出）。  
变更记录表中出现的版本号集合须与 `**versions/`** 下实际文件夹名 **一致**（无「表有目录无」「目录有表无」）。  
若某规范头部有 **「文档版本」** 字段且本次编辑需与所选迭代对齐：仅填写 `**versions/{版本号}/` 已存在** 的版本号，**不**通过变更记录表虚构新版本。
- **根目录规格的过程性留痕（`iteration_notes.md`）**  
**根目录 PRD 规格文档** 指项目根下的 `main_spec.md`、`api_spec.md`、`ui_manifest.md`、`feature_spec.md`（若项目另有其它根目录规范、且模板中含同类「变更记录」表，一并遵守下述分工）。  
**②、③、④、⑤**（含总控代跑上述步时）每次 **实际改动了上述任一文件的正文**，须在 **本次操作所使用 `版本号`** 对应的 `**versions/{版本号}/iteration_notes.md**` 中 **在文末（或文件内约定的「变更流水 / 会话记录」区）按时间顺序追加一条简短总结**（建议一行或一条列表项：日期/摘要、改了哪些文件、要点即可）。**不得** 借此 **新建** `versions/` 下更高版本文件夹、**不得** **虚构或递增文档版本号** 来「记账」；**不得** 将此类过程性留痕写入根目录各规范 **「变更记录」** 表（该表仍 **只** 由 **① 首版**、**⑧ 按新版本追加一行** 维护）。**⑥ `lny-prd-prototype`、⑦ `lny-prd-check`** 不在此列：**⑥** **不得** 修改根规格与 `**iteration_notes`**（只写原型）；**⑦** **只读**、不改任何 PRD 正文。  
`**/lny-prd-iter`** 创建 `**versions/{新版本号}/**` 时生成的 `**iteration_notes.md**` 为该版本的 **迭代开篇/结构化摘要**；同一版本目录内在 **⑧ 之外** 的多次编辑，仍可在该文件 **文末按序追加** 上述总结，与开篇内容并存、**不替代** 根目录「变更记录」里新版本行的登记职责。

# 立项 `/lny-prd-master`

## 功能

创建项目初始文档和文件夹结构，第一个版本在 `versions/v1.0.0/` 下。

> **版本纪律**：本「立项」流程 **是唯一** 允许 **初始化** `versions/v1.0.0/` 及根目录各规范中 **v1.0.0** 与 **「变更记录」首行** 的路径；新建更高版本或向根目录「变更记录」表 **追加新版本行** **仅** 见 `**/lny-prd-iter`**。其他技能 **不得** 在根「变更记录」表新增行；对根目录规格正文的编辑留痕 **仅** 允许写在 **当前版本** `versions/{版本号}/iteration_notes.md` **文末按序追加总结**，见上文 **§1.1**。

## 输入方式

仅支持 **一次性输入** 与 **Agent 对话式补充**（普通聊天消息）。

### 一次性输入（推荐）

用户在同一条消息中附上 YAML（或等价结构化说明），模型直接用于生成脚手架：

```yaml
项目名称:
产品定位:
目标用户和故事:
  - 用户: {用户A描述}
    故事: {故事A描述}
  - 用户: {用户B描述}
    故事: {故事B描述}
产品目标:
终端范围: [小程序, App, 管理后台]
竞品参考: (可选)
专业术语介绍: (可选)
```

## 框架内置能力（不纳入 PRD）

公司前后端统一框架已约定的 **通用能力** 默认 **不** 写入业务 PRD（无 `PAGE-*` / `API-*` / `FEATURE-*` / 单页 PRD / 业务原型页）。**事实源在本节**；**②③④** 及 **⑤⑥** 开笔前须 **Read 本节**，并按 **默认排除（下表）+ 本项目追加排除** 执行，**不在各子技能内重复枚举具体能力名**（后续公司框架新增能力时 **只改本节**）。

命中下表能力时，PRD **只写**「沿用框架能力 / 按框架约定配置 / 复用现有接口」；**只写**新增业务规则、字段口径、流程差异、权限差异、页面差异、状态差异。若仅为换配置项、菜单入口或页面跳转方式，**按框架约定**，**不**展开成新业务能力。确需改底层能力时，须 **单独标成「框架调整」**，**不**混写进业务 PRD。

### 仍须写 PRD 的边界

- **业务页内** 展示用户信息 → 可保留对应 `PAGE-*` + **读**接口 + 精简 Feature（**不含**鉴权/登录流程正文）。
- **业务接口** 鉴权 → `api_spec` **§1 PRD 边界** 可写 **「由公司统一框架鉴权」** 一句，**不** 展开登录接口字段。
- **支付、第三方票务** 等业务 **`EXT-*`** → **仍须**编写；PRD **只写**与本项目相关的 **业务差异**（场景、状态、校验、字段口径），**不**重写通道/SDK 通用流程。
- **业务 `API-*` / `FEATURE-*`** → 订单、列表、业务配置、含支付/通知 **业务分支** 的流程等 **照常**编号；**不**为「用户登录」「静默登录」「通用上传接口」等 **框架通用** 能力单独建档。
- **新增业务配置项 / 字典项 / 开关** → 在业务规格中 **只列新增项** 及口径，**不**重写系统配置模块本身。
- **TabBar / 路由**：`ui_manifest` §3.1 登记 **业务 Tab 页** 与 `pagePath` 属于 **差异配置**，**不**与「不展开 tabbar 实现」冲突。
- **上传 / 富文本 / 列表组件**：业务表单 **使用** 框架能力时，只写 **字段、校验、展示口径** 或表内 **特殊约束**，**不**重写组件/请求层实现。

### 排除口径（与子技能对齐，避免误读）

**「不为已排除项建 PAGE/API/FEATURE/EXT」** 指 **框架通用能力本身**（登录页、管理员 CRUD、通用上传服务、RBAC 配置页、微信 OAuth 通用接入说明等），**不**禁止业务侧 **引用** 这些能力。

| 场景 | 是否建规格 | 写法 |
|------|------------|------|
| AD 登录 / MP 静默登录 / 改密 | **否** | 不写 PAGE/API/FEATURE |
| 业务首页、列表、下单 | **是** | 正常 `PAGE-*` + `API-*` + `FEATURE-*` |
| 个人中心 / mine Tab **仅有框架默认空壳** | **否** | 不写 PAGE |
| mine Tab **含业务模块**（订单、会员卡等） | **是** | 建 `PAGE-*`，**只写业务分区** |
| 业务支付（下单唤起支付、失败重试） | **是** | `FEATURE-*` + 业务 `API-*`；支付 **通道** 写「沿用框架能力」 |
| 票务 / 第三方业务 `EXT-*` | **是** | `EXT-*` **只写业务差异** |
| 表单里用框架 upload / mp-html | **是** | 只写字段与 **特殊约束**，不写上传 SDK |

**与子技能冲突时**：以 **本节 + 「仍须写 PRD 的边界」** 为准。

### 本项目追加排除（立项 / 续跑）

- 立项 YAML 可用 **`框架内置排除`** 列出 **在默认排除之上** 的本项目项。
- 可选：在 `main_spec.md` 或 `versions/{版本号}/iteration_notes.md` 记一笔 **本项目排除摘要**（**非**立项模板强制章节）。
- **合并规则**：子技能以 **本节默认排除（上表）∪ 本项目追加** 为准；用户口头补充与 YAML 冲突时以 **最新用户确认** 为准。

| 步骤 | 要求 |

| ⑦ check | 若规格与 **本节 + 本项目追加** 重复展开框架能力 → 🔴 阻塞 |
