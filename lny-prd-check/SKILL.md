## 与总控的关系

全流程见 **`lny-prd-master/SKILL.md`**。本步为 **⑦ `/lny-prd-check`**：只读检查，**不**改仓库文件。

# 检查 `/lny-prd-check`

## 规范参考
- 编号体系：`main_spec.md` → **§2.1 编号规则说明**
- 页面与组件事实源：`ui_manifest.md` 索引 + `ui/PAGE-*.md` / `ui/COMP-*.md`
- 接口需求：`api_spec.md` §4 索引 + `api/API-*.md`
- UI清单：`ui_manifest.md` 索引 + `ui/PAGE-*.md` / `ui/COMP-*.md`（旧版兼容 `ui_manifest` 全文）
- Feature清单：`feature_spec.md` 索引 + `feature/FEATURE-*.md`
- 单页PRD：`versions/{版本号}/pages_prd/*.md`
- 原型：`prototypes/{终端}/**/*.html`

## 前置条件
- 必须已执行 `/lny-prd-master` 初始化项目

## 输入变量
```yaml
检查范围: latest (默认) 或 指定版本号如 v1.1.0
验收范围: all (默认) 或 doc-only / func-only
```

## 优先级评价标准

| 等级 | 图标 | 定义 |
|------|------|------|
| 高 | **高** | **阻塞性问题**：导致后续步骤无法执行、规格间存在矛盾、数据完整性与可追溯性受损。须优先修复。 |
| 中 | **中** | **质量问题**：影响最终交付质量或团队协作效率，但不阻塞流程推进。建议在进入下一步前修复。 |
| 低 | **低** | **优化建议**：格式规范、可读性提升、锦上添花的改进项。可择机处理，不阻塞。 |

## 一、文档性检查

对根规格文档及明细目录的规范性、完整性、一致性做只读检查，不涉及原型实现。

### 1.1 编号与格式类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 接口编号唯一性 | `api_spec.md` 中所有 API-XXX 不重复 | 修改重复编号 | 中 |
| 页面编号唯一性 | 主文档中页面编号不重复且符合格式 | 修正编号格式 | 中 |
| 页面模块编号规范性 | 模块编号格式为 `{页面编号}-{区域缩写}` | 修改区域缩写 | 低 |

### 1.2 内容规范类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| PRD meta 残留 | 根规格与 `api/` 正文不得含 `由后端Agent校准更新`、`待补充`、`占位说明` 等流程 meta | 在 `/lny-prd-api`、`/lny-prd-ui` 等步清理为业务内容或 **无** | 中 |
| API 超需求层 | `api/API-*.md` 不应含 `请求路由`、JSON 示例、HTTP/path 等实现向章节 | 建议在 `/lny-prd-api` 删实现向内容，保留业务需求 | 中 |
| 实现细节越界 | PRD 不应在 `api_spec`/`api/` 写 HTTP 方法/path/JSON 示例/响应 code 表 | 删除实现向内容 | 中 |
| 框架内置能力重复 | 业务 `ui/`/`api/`/`feature/`/`pages_prd`/原型不得为 `lny-prd-master` →框架内置能力 已排除项再写规格 | 删除重复项或更新 master 该节/本项目登记 | 高 |
| 主文档与 API 同步 | 主文档业务规则与 `api/` 接口需求无矛盾 | 对齐业务规则 | 高 |

### 1.3 结构完整性类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 文档完整性 | 根目录四规范含立项/索引必要章节（见 `lny-prd-master/SKILL.md` 立项产物） | 补充缺失章节 | 高 |
| UI清单完整性 | `ui_manifest.md` 包含必要章节 | 补充缺失章节 | 中 |
| 文件路径一致性 | PRD存放路径与"存放路径"一致 | 移动PRD到正确目录 | 中 |
| 索引-明细一致性（API/UI/Feature） | `api_spec`/`ui_manifest`/`feature_spec` 索引项均有对应文件；明细可回溯到索引 | 补建缺失文件或补全索引行 | 高 |

### 1.4 版本与台账类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 版本一致性 | 变更记录表版本集合与 `versions/` 一一对应；同版本号在单表不得多行；最新版本与目录一致 | 在 `/lny-prd-iter` 统一修订；本步不改文件 | 高 |
| 过程性留痕（建议） | 对根目录四规范的修订，宜在 `versions/{对应版本}/iteration_notes.md` 文末有按时间顺序的简短摘要 | 提醒遵守 `lny-prd-master` §1.1；本步不改文件 | 低 |
| iter 台账 ↔ pages_prd | `versions/{v}/ui_changes.md` 中新增/修改页面且 `pages_prd目标路径` 非「无需单页 PRD」「—」的每一行，对应文件须存在 | 运行 `/lny-prd-page`（⑤）按台账路径落盘 | 中 |
| iter 台账 ↔ 规格索引（建议） | `ui_changes.md`/`api_changes.md`/`feature_changes.md` 中状态待②/待③/待④的条目，应对应检查 `ui_manifest`/`api_spec` §4/`feature_spec` 是否已落实 | 按台账委派步运行 `/lny-prd-ui`/`/lny-prd-api`/`/lny-prd-feature` | 低 |

### 1.5 引用闭环类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 接口编号存在性 | PRD中引用的接口必须在 `api_spec.md` §4 中存在 | 补充或修正引用 | 高 |
| Feature 引用闭环 | 每个 FEATURE-* 的关联页面/关联接口在根规格可追溯且不孤儿引用 | 在 `/lny-prd-feature` 修复引用或补齐页面/接口 | 中 |
| 页面跳转闭环检查 | 跳转目标页面必须存在，双向可达 | 补充缺失页面 | 中 |
| 接口覆盖率检查 | 每个页面至少关联一个接口（静态页面除外） | 补充接口或标记静态 | 低 |
| 局部自定义UI组件规格 | 每个 `ui/COMP-*.md` 含设计前提与 UI 状态矩阵（必填）；`ui/PAGE-*` 未重复 COMP 内状态详述 | 在 `/lny-prd-ui` 补 COMP 状态矩阵或删 PAGE 重复状态描述 | 中 |
| 局部组件聚合与拆页 | 视觉相邻且同一状态机的区块未错误拆成多个 COMP；单页 ≥3 独立状态机/数据域时 `ui/PAGE-*` 有拆分说明 | 在 `/lny-prd-ui` 合并 COMP 或补拆页说明 | 中 |

### 1.6 其它类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 原型图文件存在性（建议） | 每个页面可有 `prototypes/{终端}/PAGE-{终端}-{序号}.html`；不作为规格阻塞项 | 运行 `/lny-prd-prototype`（⑥） | 低 |

## 二、功能性验收

以用户视角对照规格文档（② `ui/`、③ `api/`、④ `feature/`、⑤ `pages_prd/`）对原型 HTML（⑥ `prototypes/`）做功能实现度验收。只读输出评价与委派建议，不回写文件。

### 功能性验收执行步骤

1. **读取规格层**（②③④⑤）：
   - `ui/PAGE-*.md` / `ui/COMP-*.md` — ② UI 层：页面布局、组件状态矩阵
   - `api/API-*.md` / `api_spec.md` §4 — ③ API 层：接口需求、业务字段
   - `feature/FEATURE-*.md` — ④ Feature 层：业务逻辑、时序/流程图、验收标准
   - `versions/{v}/pages_prd/*.md` — ⑤ 单页 PRD：页面交互描述
2. **读取原型层**（⑥）：
   - `prototypes/{终端}/**/*.html` — 原型 HTML 文件
3. **逐层对照验收**：以用户视角逐页体验原型，对照四层规格逐一核对功能实现度。
4. **输出委派建议**（不直接委派，等待用户确认）：
   - 规格层问题（②③④⑤） → 建议先走对应步修正规格，再走 ⑥ 刷新原型
   - 仅原型实现问题（规格无问题） → 建议直接走 ⑥ 优化原型

### 2.1 原型交互验收

对照 `lny-prd-prototype/SKILL.md` 壳层规则，检查原型 HTML 的交互合规性。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 原型壳层 COMP 状态演示 | 汇总页/桌面壳：§C 面板固定宽 200px（收起后 0）；顶栏 §C 收起；状态选项与 `ui/COMP-*` 矩阵一致 | 建议运行 `/lny-prd-prototype` 按 §C 补壳层控制 | 中 |
| 壳层规格 Drawer 宽度 | 右侧规格说明 Drawer 固定 375px（与设备逻辑宽一致） | 建议运行 `/lny-prd-prototype` 按 §D §7 调整 Drawer | 中 |
| 移动端预览区等比缩放 | MP/H5/APP 汇总页：`preview-area` 无外围滚动条；含 `phone-scale-host` + `fitPhoneFrame()`；页单/§C 顶栏收起、切页 §C 显隐、resize 时重算 | 建议运行 `/lny-prd-prototype` 按 §D 补壳层缩放与 §C 顶栏收起 | 高 |
| 原型演示按钮归位 | 各 `PAGE-*.html` 内不得含 `demo-*`/`setDemo*`/`mock-*` 等演示专用按钮（后端状态切换须归位 §C）；页内可保留用户操作按钮 | 建议运行 `/lny-prd-prototype` 按 §C.5+§G.2 移至 §C，删页内演示按钮 | 中 |
| 原型 JS 引用完整性 | 原型 HTML 打开 F12 Console 无 ReferenceError/TypeError；搜索被删/重命名函数/变量确认无残留引用 | 建议运行 `/lny-prd-prototype` 按 §G.1 修复引用断裂 | 高 |
| 原型 §C 状态选项同步 | §C 面板每个 COMP 的状态切换选项与 `ui/COMP-*.md` 的 UI 状态矩阵逐字一致；不多不少不重名 | 建议运行 `/lny-prd-prototype` 按 §G.3.4 对齐选项与 COMP 矩阵 | 中 |

### 2.2 UI 规格验收（对照 ②）

以 `ui/PAGE-*.md` / `ui/COMP-*.md` 为基准，验收原型 UI 实现度。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 页面布局与原型一致 | 原型页面布局结构与 `ui/PAGE-*.md` 描述一致；模块区域、嵌套关系无偏差 | 建议走 `/lny-prd-ui` 修正布局描述，或 `/lny-prd-prototype` 同步原型 | 中 |
| UI 组件状态覆盖 | 原型中 COMP 状态切换与 `ui/COMP-*.md` 状态矩阵全部对应；所有态视觉可演示 | 建议走 `/lny-prd-ui` 补状态矩阵，或 `/lny-prd-prototype` 补状态演示 | 中 |

### 2.3 API 规格验收（对照 ③）

以 `api/API-*.md` / `api_spec.md` §4 为基准，验收原型数据交互实现度。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 接口字段与原型匹配 | 原型页面展示/交互的数据字段能在 `api/API-*.md` 中找到对应业务字段定义；无遗漏、无多余 | 建议走 `/lny-prd-api` 补字段定义，或 `/lny-prd-prototype` 调整原型数据 | 中 |
| 接口覆盖交互场景 | 原型中每个用户操作触发的数据交互，均有对应 `API-*` 接口需求支撑 | 建议走 `/lny-prd-api` 补接口需求，或 `/lny-prd-prototype` 调整交互逻辑 | 中 |

### 2.4 Feature 流程验收（对照 ④）

以 `feature/FEATURE-*.md` 为基准，验收原型业务流程实现度。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 业务逻辑与原型一致 | 原型操作流程与 `FEATURE-*` 描述的业务规则、前置条件、分支逻辑一致 | 建议走 `/lny-prd-feature` 修正流程描述，或 `/lny-prd-prototype` 同步原型 | 高 |
| 验收标准原型可达 | `FEATURE-*` 中每条验收标准在原型中均可逐一验证通过 | 建议走 `/lny-prd-feature` 调整验收标准，或 `/lny-prd-prototype` 补充交互 | 中 |

### 2.5 单页 PRD 验收（对照 ⑤）

以 `versions/{v}/pages_prd/*.md` 为基准，验收单页交互描述与原型一致性。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 页面交互与 PRD 一致 | 原型交互细节与 `pages_prd/*.md` 描述的交互流程、边界情况一致 | 建议走 `/lny-prd-page` 修正描述，或 `/lny-prd-prototype` 同步原型 | 低 |

### 2.6 跨层一致性验收

检查规格层（②③④⑤）与原型层（⑥）之间的引用闭环与状态覆盖完整度。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 跨层引用闭环 | 原型中涉及的 PAGE-* / API-* / FEATURE-* 编号均可回溯到对应规格索引，无孤儿引用 | 建议定位缺失环节后走对应步补建引用 | 高 |
| Feature 状态覆盖完整度 | `FEATURE-*` 中定义的所有业务状态（正常/异常/边界/空）在原型中均有对应页面状态可演示 | 建议走 `/lny-prd-prototype` 补全状态演示 | 中 |

### 2.7 用户体验评价

以终端用户视角评价原型的整体可用性。

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 页面跳转流畅度 | 原型页面间跳转自然，无死循环或跳转失败；导航路径清晰 | 建议走 `/lny-prd-prototype` 修复跳转逻辑 | 低 |
| 数据展示合理性 | 原型中数据展示格式、数量、排列与实际业务预期一致，无明显不合理 | 建议走 `/lny-prd-prototype` 调整数据展示 | 中 |
| 交互细节完整性 | 按钮/表单/列表/弹窗等交互组件行为符合预期；hover/active/disabled 等状态可演示 | 建议走 `/lny-prd-prototype` 补交互细节 | 中 |

### 2.8 其它类

| 检查项 | 规则 | 不通过建议 | 优先级 |
|--------|------|------------|--------|
| 其它功能性验收 | 以上分类未覆盖的功能性差异或体验问题 | 视问题归属给出对应步委派建议 | — |

## 委派建议输出格式

功能性验收的委派建议按以下格式输出，等待用户确认后再执行：

```
## 委派建议

### 规格层修正（先执行）
- [② UI] 问题描述 → 建议运行 `/lny-prd-ui` 修正 xxx
- [③ API] 问题描述 → 建议运行 `/lny-prd-api` 修正 xxx
- [④ Feature] 问题描述 → 建议运行 `/lny-prd-feature` 修正 xxx
- [⑤ Page] 问题描述 → 建议运行 `/lny-prd-page` 修正 xxx

### 原型层同步（待规格修正后执行）
- [⑥ Prototype] 问题描述 → 建议运行 `/lny-prd-prototype` 同步原型

---
是否确认以上委派建议？输入「继续」逐一执行，或指定跳过项。
```

## 执行范围

- **`检查范围: latest`（默认）**：`versions/` 下 **semver 最大** 的版本目录；检查根目录 **`main_spec.md`、`api_spec.md`、`ui_manifest.md`、`feature_spec.md`**，以及该版本下 **`pages_prd/`**、**`prototypes/`**（含 `versions/{版本号}/prototypes/` 镜像若存在），并核对 `api/`、`ui/`、`feature/` 明细目录（若启用目录化模式）；若存在 **`ui_changes.md` / `api_changes.md` / `feature_changes.md`**，执行 **iter 台账** 相关检查项。
- **指定版本号**：同上，但限定为 `versions/{版本号}/`。

## 执行步骤
1. 读取根目录 `main_spec.md`、`api_spec.md`、`ui_manifest.md`、`feature_spec.md`（若存在）
2. 若指定版本号，额外读取 `versions/{版本号}/` 下文件
3. **文档性检查**：执行第一大类所有检查项（含目录化索引-明细一致性、Feature 引用闭环）
4. **功能性验收**：读取原型 HTML 及对应 ②③④⑤ 规格文件，逐层对照验收
5. 输出按 **检查性质 → 优先级** 排列的检查报告；功能性验收问题附委派建议
